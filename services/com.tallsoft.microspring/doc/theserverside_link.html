MicroSpring
A Spring Framework (mainly) compatible IOC container by J.Gibbons, Feb 2005
http://www.tallsoftware.com/microspring/

The Server Side have kindly linked to MicroSpring. Here I try to explain very briefly the reasoning that led to MicroSpring.

First, a bow to the Spring Framework team, who have produced a great framework, and if you need a total solution it's where you should go. However, my perspective is that I need a smaller implementation and do not need many of it's features. Also an IOC framework should be simple to learn, a page or two of documentation at most.

I have not yet talked to the Spring team, I am not endorsed by the Spring Framework, and I am not competing in some rabid open source slanging match. I just want a smaller code base, and no 3rd party library dependencies. Now, on to the kernel of the matter.

My Requirements

To replace existing in-house, context based xml configurations with an IOC, test friendly framework. The solution to be used on many small to medium sized projects. Modules/sub-system running within the framework to be micro-bootable.

The Assertion

IOC is not hard, the Spring Framework has become the prominant implementation. However, the Spring Framework does far more than IOC, and is now the size of an alternative J2EE, with hooks to many other currently popular open source technologies. It is micro-bootable through the refresh functions on the ApplicationContexts (bean factories) - but I really need direct control of this.

Also, JDK1.5 is powerful, cohesive, comprehensive, supported, free, and you don't risk pulling in terror-ware from the net. So an IOC framework based upon this should need nothing else.

The Proposal

To strip away all the non IOC parts of Spring. Not only that, but to remove all the customisations of IOC within Spring, to leave a single small implementation. One with few lines of code, and zero dependencies on other open source. To embrace JDK1.5 in terms of new core libs and the new language features.

The solution

To write a new IOC container. One which is compatible with Spring, but a vastly smaller framework. Focused purely on IOC, with no integration with, or dependency on other open source projects. The reference implementation to be 'hacked out' over a weekend due to time constraints.

The result

MicroSpring! And it's done, and tested, and released! Spring IOC without Spring. But also, a challenge to others to produce an even smaller and leaner implementation of IOC, which is ideally Spring compatible. This should be possible given more time, and if I find more, I may make MicroSpring smaller, leaner, and, err, meaner?

So, if you want to play with IOC, remain compatible with Spring, and thats all, read on, and play with MicroSpring.
